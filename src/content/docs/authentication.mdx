ConvoMem is built to hold sensitive customer data responsibly. This guide covers how you
authenticate and how your data is protected.

## API keys

Every request is authenticated with an `X-API-Key` header:

```typescript title="request.ts"
const res = await fetch('https://api.convomem.com/api/v1/customers?limit=1', {
  headers: {
    'X-API-Key': process.env.CONVOMEM_API_KEY!,
  },
})
```

The SDK handles auth automatically — pass your key at initialization:

```typescript title="init.ts"
import { ConvoMem } from '@optraze/convomem-sdk'

const client = new ConvoMem({
  apiKey: process.env.CONVOMEM_API_KEY!,
})
```

<Callout type="note" title="No Bearer prefix">
ConvoMem uses `X-API-Key` header authentication, not `Authorization: Bearer`.
</Callout>

## Scopes

Organization API keys (`sk-org-*`) have **integration scope** and can access:

- Capture (`POST /capture`)
- Customer management (`/customers/*`)
- Memory management (`/customers/*/memories`)
- Conversations (`/customers/*/conversations`)
- Handoff briefings (`/customers/*/handoff`, `/customers/handoff`)
- Embed tokens (`/embed/tokens`)

Administrative endpoints — webhooks management, org settings, audit logs, and analytics
dashboards — require session-based authentication through the dashboard. They **cannot**
be accessed with an org API key.

<Callout type="danger" title="Rotate compromised keys immediately">
If a key leaks, revoke it from the dashboard. Revocation takes effect within seconds and
cannot be undone.
</Callout>

## PII redaction

Before any memory is persisted, ConvoMem runs detection for emails, phone numbers, card
numbers, and other sensitive patterns, replacing them with typed placeholders. The
original values are never written to long-term storage.

## Data residency

Choose where your data lives at the project level. On-prem and single-tenant deployments
are available for regulated workloads — see the [changelog](/changelog) for the latest
availability.
